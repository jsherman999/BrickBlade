Code overview

BrickBlade

BrickBlade is a personal LEGO collection backend. It lets a client scan or type a set, identify a box photo, look up catalog metadata, fetch sealed and used market prices, store owned inventory, and keep historical price snapshots for collection value tracking.

Python 3.12 FastAPI Typer CLI SQLAlchemy SQLite WAL httpx launchd jobs Rebrickable CSV mirror

What the app does

High-level behavior

The app is a backend service plus a matching command-line tool. The FastAPI server exposes inventory, lookup, photo-identification, and refresh endpoints, while the Typer CLI exposes the same core behaviors for local administration and smoke tests.

The lookup path starts with either a set number, a UPC/EAN barcode, or an uploaded image. Once BrickBlade has a set number, it normalizes it into the Rebrickable/BrickLink number-variant format, reads set metadata from the local SQLite catalog mirror, and returns the freshest known prices.

Runtime stack

Underlying services and software

  • No Redis, Celery, Postgres, Django, frontend build system, or long-running worker queue is used.
  • HTTP API uses FastAPI served by uvicorn, usually on port 8765.
  • CLI uses Typer through the brickblade console script.
  • Persistent data is SQLite by default at var/brickblade.db, configured by BRICKBLADE_DB_URL.
  • Database access uses SQLAlchemy; SQLite runs with WAL, foreign keys, and normal synchronous mode.
  • HTTP clients use httpx; BrickLink signing uses OAuth1 via oauthlib.
  • Recurring production tasks are macOS launchd agents, not an in-app scheduler.
  • Configuration comes from environment variables and .env through pydantic-settings.

Call flow

Main lookup path

Client entry

iPhone client, curl, or CLI calls /api/lookup or brickblade lookup.

→

Auth and dependencies

api.deps verifies the bearer token, opens a DB session, and creates upstream clients.

Resolve set number

core.lookup.resolve_barcode() asks Brickset first, then UPCitemdb when only a barcode is provided.

→

Read catalog mirror

core.sets.get_metadata() joins catalog_sets to catalog_themes in local SQLite.

Fetch or reuse prices

core.pricing.get_or_refresh() uses cached snapshots until the TTL expires, then calls Brickset and BrickLink.

→

Return result

The API and CLI receive one LookupResult: normalized set number, metadata, prices, and source list.

Core service functions

Important internal behavior

  • normalize_set_num() converts 10294 to 10294-1 but leaves explicit variants intact.
  • lookup_set() is the shared API/CLI path, so behavior stays aligned across both interfaces.
  • get_or_refresh() treats the prices table as both cache and history; refreshes append new rows rather than overwriting old rows.
  • session_scope() commits successful operations and rolls back failed ones, including API requests and job chunks.
  • request_with_retry() retries transient network problems, 429s, and 5xx responses for all external clients.
  • refresh_prices.run() staggers requests between owned sets to be polite to upstream APIs.

Data model

SQLite tables

catalog_sets

Local Rebrickable set catalog: set number, name, year, theme id, part count, and image URL.

catalog_themes

Theme hierarchy used to show friendly theme names with set metadata.

catalog_parts, catalog_colors

Part and color reference data mirrored from Rebrickable for future part-level inventory work.

catalog_inventories

Rebrickable inventory version rows connecting sets to their component inventory records.

catalog_inventory_parts

Part quantities, colors, spare flags, and image URLs for each mirrored set inventory.

catalog_inventory_sets, catalog_inventory_minifigs

Sub-set and minifigure contents for larger set inventories.

inventory

BrickBlade-owned rows: set number, quantity, condition, notes, and creation timestamp.

prices

Append-only price history with source, condition, guide type, currency, average/min/max, quantity, raw JSON, and fetch time.

catalog_import_log

CSV import diagnostics and idempotency: file name, row count, sha256 digest, and import timestamp.

Jobs

Catalog and price refreshes

  • brickblade import-catalog downloads Rebrickable .csv.gz files from their CDN.
  • Each CSV is hashed with sha256; unchanged files are skipped unless --force is used.
  • Changed catalog tables use a replace-all import strategy inside a transaction, then archive the raw CSV under var/rebrickable-archive/<date>.
  • brickblade refresh-prices finds distinct owned set numbers, forces lookups, and appends fresh price rows.
  • launchd/install.sh installs an always-on API agent, a Sunday 03:00 catalog refresh, and a nightly 04:00 price refresh.

Configuration

Environment variables

  • BRICKBLADE_BEARER_TOKEN: shared API token sent as Authorization: Bearer ....
  • BRICKBLADE_DB_URL: SQLAlchemy database URL, defaulting to sqlite:///./var/brickblade.db.
  • BRICKBLADE_DATA_DIR: directory for downloads, archives, logs, and default SQLite storage.
  • BRICKBLADE_PRICE_TTL_HOURS: cache freshness window before pricing calls are made again.
  • BRICKSET_KEY and optional BRICKSET_USERNAME/BRICKSET_PASSWORD: Brickset API access.
  • BL_CONSUMER_KEY, BL_CONSUMER_SECRET, BL_TOKEN, BL_TOKEN_SECRET: BrickLink OAuth1 credentials.
  • REBRICKABLE_KEY: configured for future API calls; the current catalog mirror uses public CSV downloads.

What talks to the network

Caller Service Used for Fallback behavior

UpcItemDbClient.find_lego_set_number() UPCitemdb trial API Barcode fallback when Brickset does not resolve the code. Returns no match if the title is not LEGO or no set number can be parsed.
BrickognizeClient.predict_set() Brickognize API Image recognition for box-photo identification. Request failure propagates to the API as an error response.

CLI

Local commands

  • brickblade health: prints paths and whether API credentials are configured.
  • brickblade init-db: creates tables idempotently.
  • brickblade import-catalog: downloads and imports Rebrickable CSV data.
  • brickblade refresh-prices: refreshes all owned-set price snapshots.
  • brickblade add, remove, list: manage local inventory.
  • brickblade lookup: lookup by set number or barcode from the terminal.
  • brickblade value: sums latest known average prices across owned sets.
  • brickblade serve: starts the FastAPI server in the foreground.

Testing

How behavior is verified

  • tests/test_core.py covers set-number normalization, barcode resolution, cache TTL behavior, forced refreshes, and lookup orchestration.
  • tests/test_api.py uses FastAPI dependency overrides so API tests do not call real upstream services.
  • tests/test_import_catalog.py exercises CSV import behavior with controlled data.
  • tests/test_clients.py checks client request construction and retry-adjacent behavior using fake transports.
  • Most tests use temporary SQLite databases by overriding BRICKBLADE_DB_URL and clearing cached settings.

Under the covers

Operational notes

The local Rebrickable mirror is the app's source of truth for catalog metadata. Live APIs are used for identification and pricing, but a set still has to exist in the local mirror before lookup_set() returns a successful result.

  • API auth is intentionally simple: every protected route compares the request header to one configured bearer token.
  • Database tables are created automatically at app startup and by setup/job commands via create_all().
  • Price data is never updated in place; each fetch appends a row so value history comes for free.
  • The default deployment expects a Mac mini reachable from the iPhone over Tailscale.
  • The SwiftUI iPhone client described in the README is not part of this repo yet; this repo is the backend and CLI.
